# Discord Webhook Task Scheduler

A full-stack web application for creating, managing, and monitoring scheduled tasks that send messages to Discord channels via webhooks.

## Features

- 📅 **Cron-based task scheduling** — powered by `node-cron`
- 🔁 **Retry logic** — exponential backoff via BullMQ + Redis
- 🔐 **JWT Authentication** — access token (15m) + refresh token (7d) stored in DB
- 📊 **Dashboard** — live summary of active, paused, and failed tasks
- 📋 **Execution logs** — per-task log viewer with auto-refresh
- 🐳 **Fully containerised** — Docker Compose with 4 services

## Running Locally

### 1. Clone and install dependencies

```bash
# Install backend dependencies
cd backend
npm install

# Install frontend dependencies
cd ../frontend
npm install
```

### 2. Configure environment variables

**Backend** — copy and edit `backend/.env.example`:
```bash
cp backend/.env.example backend/.env
```

```env
DATABASE_URL=postgresql://postgres:password@localhost:5432/discord_scheduler
REDIS_URL=redis://localhost:6379
ACCESS_TOKEN_SECRET=your_access_secret_here
REFRESH_TOKEN_SECRET=your_refresh_secret_here
PORT=4000
NODE_ENV=development
```

**Frontend** — copy and edit `frontend/.env.example`:
```bash
cp frontend/.env.example frontend/.env.local
```

```env
NEXT_PUBLIC_API_URL=http://localhost:4000
```

### 3. Run database migrations

```bash
cd backend
npx prisma migrate dev --name init
```

This auto-generates the `users`, `tasks`, and `task_logs` tables — no manual SQL required.

### 4. Start the backend

```bash
cd backend
npm run dev
# API available at http://localhost:4000
# Health check: http://localhost:4000/health
```

### 5. Start the frontend

```bash
cd frontend
npm run dev
# App available at http://localhost:3000
```

### Step-by-step deployment

**1. Upload project files to the VPS**

Option A — using `scp` from your local machine:
```bash
scp -r ./insignia-technical-test user@YOUR_VPS_IP:/home/user/app
```

Option B — clone from Git (recommended):
```bash
ssh user@YOUR_VPS_IP
git clone https://github.com/your-username/your-repo.git app
cd app
```

**2. Create and configure `.env`**

```bash
cd app
cp .env.example .env
nano .env   # or: vim .env
```

Set every value — especially:

```env
POSTGRES_PASSWORD=super_strong_production_password
ACCESS_TOKEN_SECRET=<64-char random hex>
REFRESH_TOKEN_SECRET=<64-char random hex>

# Replace with your VPS public IP or domain:
NEXT_PUBLIC_API_URL=http://YOUR_VPS_IP:4000
```

> ⚠️ `NEXT_PUBLIC_API_URL` is baked into the Next.js bundle at **build time**.  
> It must be the URL your **browser** uses to reach the backend — not the internal Docker network address.

**3. Open firewall ports**

```bash
# Allow frontend and backend ports
sudo ufw allow 3000/tcp
sudo ufw allow 4000/tcp

```

```bash
# Pull latest images and start — no build required
docker compose pull
docker compose up -d
```

Or if you want to build from source instead:

```bash
docker compose up --build -d
```

The `-d` flag runs everything in the background (detached).

**5. Verify services are running**

```bash
docker compose ps
docker compose logs backend --tail=50
curl http://localhost:4000/health
```

**6. Access the app**

| URL | Service |
|-----|---------|
| `http://YOUR_VPS_IP:3000` | Frontend |
| `http://YOUR_VPS_IP:4000` | Backend API |

### (Optional) Nginx reverse proxy with a domain

If you own a domain (e.g. `app.yourdomain.com`), use Nginx to serve both services from port 80/443 instead of exposing raw ports.

**Install Nginx:**
```bash
sudo apt install nginx -y
```

**Create config** at `/etc/nginx/sites-available/scheduler`:
```nginx
server {
    listen 80;
    server_name app.yourdomain.com;

    # Frontend
    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }

    # Backend API
    location /api/ {
        proxy_pass http://localhost:4000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

**Enable and reload:**
```bash
sudo ln -s /etc/nginx/sites-available/scheduler /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
```

**Update `.env`** so the frontend calls the API through the same domain:
```env
NEXT_PUBLIC_API_URL=http://app.yourdomain.com/api
```

Then rebuild:
```bash
docker compose up --build -d frontend
```

**Add SSL with Certbot (HTTPS):**
```bash
sudo apt install certbot python3-certbot-nginx -y
sudo certbot --nginx -d app.yourdomain.com
```

After SSL, update `.env` again:
```env
NEXT_PUBLIC_API_URL=https://app.yourdomain.com/api
```

## API Endpoints

Base URL (local): `http://localhost:4000`  
Base URL (production): `https://api-insignia-tech-intv.sizu.dev`

All protected routes require the header:
```
Authorization: Bearer <accessToken>
```

### Unified Response Format

All API responses follow a consistent shape:

| Shape | When used |
|-------|-----------|
| `{ success, data, meta }` | List endpoints (paginated) |
| `{ success, data }` | Single resource |
| `{ success, message, data }` | Create / update actions |
| `{ success, message }` | Delete / logout (no resource returned) |
| `{ success: false, error: { code, message } }` | All errors |

**Error codes:** `UNAUTHORIZED` · `NOT_FOUND` · `VALIDATION_ERROR` · `CONFLICT` · `INTERNAL_ERROR`

### Step 4 — Bug Fixes During Development

After running the application, several issues were caught and fixed through follow-up messages:

**TypeScript JWT error**
> *"return new TSError... Type 'string' is not assignable to type 'StringValue'"*

The `expiresIn` field in `jsonwebtoken` v9 requires a branded `StringValue` type, not a plain `string`. Fix: added `as const` to both token expiry values in `backend/src/config/env.ts`.

**Redis connection crash**
> *"AggregateError: connect ECONNREFUSED 127.0.0.1:6379"*

The app crashed on startup when Redis was not running locally. Fix: added `lazyConnect: true` and a `retryStrategy` to the ioredis config, and wrapped `createWebhookWorker()` in a try/catch so the app starts gracefully without Redis.

**React hydration mismatch**
> *"A tree hydrated but some attributes of the server rendered HTML didn't match..."*

Three separate root causes were found and fixed:
1. `app/page.tsx` and `app/(dashboard)/page.tsx` both mapped to `/` in Next.js App Router — deleted the conflicting file.
2. A `mounted` guard in the dashboard layout was dropping `{children}` from server HTML, causing RSC payload mismatch — removed the guard entirely.
3. `toLocaleString()` produces different output on server vs client — replaced with a `<ClientDate>` component that uses `suppressHydrationWarning`.

## Database Schema

Defined in `backend/prisma/schema.prisma` — **no manual SQL**.

```
User ──< Task ──< TaskLog
```

| Table | Key Fields |
|-------|-----------|
| `users` | `id`, `username`, `passwordHash`, `refreshToken`, `createdAt` |
| `tasks` | `id`, `userId`, `name`, `schedule`, `webhookUrl`, `payloadJson`, `maxRetry`, `status` |
| `task_logs` | `id`, `taskId`, `executionTime`, `status`, `retryCount`, `message` |

**Run migrations:**
```bash
# Development (interactive, creates migration files)
cd backend && npx prisma migrate dev --name init

# Production / Docker (non-interactive)
cd backend && npx prisma migrate deploy

# View DB in Prisma Studio
cd backend && npx prisma studio
```
